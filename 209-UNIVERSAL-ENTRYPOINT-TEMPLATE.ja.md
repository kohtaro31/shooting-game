# 209 — ユニバーサル・エントリポイント（テンプレート）
## AIと開発するための「コンテキスト温存リポジトリ」運用テンプレ

この文書は、どんな新規プロジェクトにもコピーして使える**テンプレート**です。

AI支援開発で最も起きがちな失敗は、コードの正誤そのものではなく、
**「それっぽいが、前提（コンテキスト）が欠けたまま／ズレたまま進む」**ことです。

これは仕様書ではありません。
方位磁針（orientation）と、作業協定（operating agreement）です。

このテンプレが存在する理由（origin）:

- 長期のAI協働では、最大の失敗は「間違ったコード」ではなく**コンテキスト喪失**です。
- チャットは揮発し、決定・制約・前提はセッションやAI個体を跨いでズレます。
- 少数の“耐久文書”があると、**安全に中断・再開**でき、説明のし直しと作り直しが減ります。
- 毎回「自然発生で発明し直す」ことを避けるため、この環境づくりを標準化します。

---

## 0. TL;DR（AI向け）

コードを書く前に:

1) `docs/*ENTRYPOINT*.md` を読む（本書またはプロジェクトの入口）
2) `docs/*ACTIVE-CONTEXT*.md` を読む（「今日」有効な前提）
3) `docs/*SCOPE*.md` と `docs/*GOALS*.md` を読む（成功条件と境界）
4) `docs/*DECISIONS*.md` を読む（決定と理由）
5) 曖昧なら **止まって質問する**

ルール:

- 古いメモや雰囲気から要件を“補完”しない。
- 小さく・可逆で・差分がレビュー可能な変更を優先する。
- 意味のある決定は、チャットではなく文書に残す。
- 文書作成の言語方針: **日本語をデフォルト**とする（識別子/コマンド等はバッククォートで保持。精度のために英語が必要な場合のみ英語を混ぜる）。

## 1. このリポジトリが何か

プロジェクト名: **SSLab — ゲーム開発の部屋**

- 何を作るか: ブラウザで動くシングルファイルHTMLゲーム（縦スクロールシューター系）
- 誰のためか: コーディング初心者のユーザー自身。作りながら学ぶことが目的。
- なぜ今か: ゲーム開発を通じてコーディング・セキュリティを学び、209文書を実践でブラッシュアップするため。
- 絶対に守る制約:
  - 単一HTMLファイル完結（外部依存なし）
  - Canvas 2D API・純粋なHTML/JS/CSS のみ
  - 変更は小さく・可逆・レビュー可能に保つ
  - セキュリティ上の理由は必ず言語化して説明する

このリポジトリは、次の最適化を優先します:

- 継続性（continuity）
- 復元性（recoverability）
- 意図の明確さ（clarity of intent）

コードは重要です。
しかし **コンテキストはコードより長生きします**。

## 2. 最小の文書システム（「4つの文書」）

AI支援開発は速い一方で、コンテキストが簡単にリセットされます。
チャットにすべてを記憶させるのではなく、「重要なこと」を少数の文書に固定します。

この“システム”は約束ではありません。
完成された方法論というより、ジャズのように育つ実験です。

まずは次の4文書を作り、維持してください:

1) **アーキテクチャ概要（Architecture Overview）**
   - 主要コンポーネントと境界は何か？
   - データは何で、どこを流れるか？
   - 来週、記憶喪失でも再開できるために何が必要か？

2) **技術的考慮（Technical Considerations）**
   - 制約、危険地帯、既知の落とし穴、性能/セキュリティ注意点
   - 「繰り返すと最も腹が立つこと」は何か？

3) **作業手順（Workflow Process）**
   - 実際に機能している手順（今週うまくいったやり方）
   - 起動・テスト・レビュー・リリース手順
   - 人間↔AIの協業の仕方

4) **ストーリー分解（Story Breakdown）**
   - 小さく、検証可能な“ストーリー”に切る
   - 各ストーリーは短いセッションで終わり、レビュー可能な差分を生む

この4つは「完璧な方法論」ではありません。
**記憶装置（memory system）**です。

---

## 3. 文書レイヤ（推奨）

分類体系は任意です。
ただし「どれが権威で、どれが参考か」という振る舞いは必須です。

推奨レイヤ:

- **Foundations**: 非交渉の前提（セキュリティモデル、脅威モデル、不変条件）
- **Orientation**: 作業の入口（本書、読む順序）
- **Active Context**: いま正しいこと（現在フェーズ、アクティブ仕様）
- **Decisions**: なぜXを選び、Yを捨てたか
- **Runbook**: 起動/ビルド/テスト/デバッグ
- **History**: 実験、却下案、行き止まり（連続性のために保持。権威ではない）

権威ルール:

- Foundations > Active Context > Decisions > Runbook > History
- Historyは参照であって命令ではない。

## 4. 協業プロトコル（人間 + AI）

AIは「高速だが記憶が不完全な協力者」として扱います。
狙いは“完璧な1プロンプト”ではありません。
狙いは、時間が経っても崩れにくいフィードバックループです。

### 4.1 Gate A — まず質問（Questions First）

新しいタスク開始時、AIは不足している制約を質問します。
人間は文書化できる形で答えます。

重要: 良い質問を作るのは難しいので、AIは**前提（仮説）も明示**します。

最小の質問セット（アプリ種別に依存しない）:

1) 望む成果と受け入れ条件（Acceptance Criteria）は何か？
2) 明確にスコープ外のものは何か？
3) 絶対に破ってはいけない制約は何か（セキュリティ、データ取り扱い、互換性など）？
4) 統合ポイントはどこか（API、DB、ファイル、外部サービス）？
5) 失敗したときのロールバック手段は何か？

仮説提示テンプレ（AIは必ずやる）:

- 「このタスクを進めるにあたり、私はX/Y/Zを仮定しています。正しいですか？」
- 「想定する境界条件（`null` / タイムアウト / 権限拒否 など）はこう扱うつもりです。受け入れ可能ですか？」

### 4.2 Gate B — 小さなストーリーで進める

作業は“ストーリー”単位で行います。ストーリーは:

- 小さい（レビュー可能）
- テスト可能（確認手順がある）
- 可逆（影響範囲が小さい）

検証できないストーリーは、まだ“準備不足”です。

実務ガイド:

- ストーリーは、**検証時間**が概ね15〜30分に収まるように切る。
  （目的はスピードではなく、コンテキストが破綻しないサイズに保つこと）

### 4.3 Gate C — 非同期ループ（Time Dilation）

よくある形:

1) 人間が次のストーリーを投げる
2) AIが実装する（人間は別作業）
3) 人間が短時間でテストし、修正点を返す
4) 繰り返す

この形が成立する条件は、文書がターン間の“共有メモリ”であることです。

### 4.4 Gate D — 先に検証、次に差分

実装後:

- 必要なチェック（テスト/リント/ビルド等）を走らせる
- 「何が変わったか」「触ったファイル」「どう検証したか」を報告する

加えて、AIは文書更新案を**差分として提示**します:

- `Decisions` / `Active Context` / `Runbook` / `Technical Considerations` への「追記候補（Doc update candidates）」
- 人間は `approve / edit / reject` で応答する

### 4.5 Gate E — コミット規律（人間主導）

- コミットのタイミングと境界は人間が決める。
- AIは変更を“マイクロステップ”でコミット可能な形に整える。

## 5. コンテキスト腐敗を防ぐ

コンテキストが腐るのは、決定がチャットにだけ存在するときです。
それを防ぐ単純ルール:

**明日も重要なら、今日書く。**

運用ルール:

- 挙動を変えたら `Active Context` と/または `Runbook` を更新する。
- 選択肢から選んだなら `Decisions` に残す。
- 落とし穴を見つけたら `Technical Considerations` に追記する。
- 同じ指示を繰り返すなら `Workflow Process` に移す。

ボトルネック対策（推奨）:

- 毎回、人間がゼロから文書を書く前提にしない。
  **AIが文書追記案を下書きし、人間は主にレビュー/承認する**。

文書は、未来の混乱した自分へのメッセージです。
未来の自分（と未来のAI）は、また初心者だと仮定します。

---

## 5.1 スケール注意（文書が巨大化したら）

Markdownの静的文書は意外と遠くまでスケールしますが、無限ではありません。
AIが毎回“新規に全部読む”のが難しくなったら、軽量の取得補助を足します:

- `docs/INDEX.md`: 「何があるか」と「どこから読むか」の地図
- `docs/SUMMARY.md`: 現在の真実を1〜2ページで要約（定期更新）
- タグ/リンク: タスク別の“読む束”を作る（認証、暗号、UI、デプロイ等）

上級編（任意）:

- 文書を構造化（エンティティ＋関係）して検索性を上げ、
  ストーリー開始時に「そのタスクに必要な分だけ」コンテキスト注入する（ナレッジグラフ等）。

## 6. ストーリーテンプレ（コピペ用）

`Story Breakdown`（またはIssue）に、各ストーリーをこの形式で書く:

### Story: <短いタイトル>

- Goal（目的）:
- Non-goals（やらないこと）:
- Acceptance criteria（受け入れ条件）:
- Constraints（制約）:
- Implementation notes（任意）:
- Verification steps（検証手順）:
- Rollback plan（ロールバック）:
- Files expected to change（変更予定ファイル）:

- Date（日付）:

- Rationale（理由）:

## 8. 迷ったとき（人間とAI向け）

不安なら:

1) EntryPointに戻る
2) FoundationsとActive Contextを再確認する
3) 推測せず、質問する

このリポジトリは、いつでも安全に”再入場”できるべきです。

## 9. SSLab 固有の運用ポリシー（2026-04-26 採用）

このセクションは、SSLab作業場における実際の運用から生まれたポリシーです。
テンプレートの抽象的なルールを、この部屋の具体的な約束として定めます。

### 9.1 文脈ファイルの管理

- **文脈ファイルはすべて `SSLab/docs/` に置く。**
- `.claude/` フォルダには文脈データを書かない（リダイレクトポインタのみ）。
- セッション開始時の読み順:
  1. `docs/ACTIVE-CONTEXT.md`
  2. `docs/PROJECT-GOALS.md`
  3. `docs/USER-PROFILE.md`
  4. `docs/AI-WORKFLOW.md`

### 9.2 ユーザーへの説明スタイル

- 説明には必ず「なぜそう書くか」を添える。
- 専門用語はその都度かみ砕いて説明する。
- セキュリティの注意点は「怖がらせず・理由を言語化」して伝える。

### 9.3 作業の進め方

- 一度の変更は小さく、ブラウザで動作確認できるサイズに保つ。
- 実装後は「何が変わったか」と文書更新案（差分）を提示し、ユーザーのapprove/edit/rejectを得る。
- コミットのタイミングはユーザーが決める。

### 9.4 このポリシーの更新条件

- ユーザーから明示的な変更依頼があったとき。
- 同じ指示を2回以上繰り返されたとき（ = ポリシーに昇格するサイン）。
